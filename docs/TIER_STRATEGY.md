# Tier strategy

GitHub does not expose a stable achievement-threshold API. Tier numbers found
in community references should be treated as hypotheses, not product contracts.

## A professional progression

1. Create the minimum evidence for the base achievement.
2. Wait for the profile to update before scaling the experiment.
3. Verify whether private evidence counts for that specific achievement.
4. Increase one tier at a time and record the observed result.
5. Stop when additional activity would make the public project less useful.

## Community-observed checkpoints

These values are included as experiment checkpoints only:

| Achievement | Checkpoints |
|---|---|
| Pull Shark | 2, 16, 128, 1024 merged authored PRs |
| Pair Extraordinaire | 10, 24, 48, 64 co-authored commits |
| Galaxy Brain | 2, 8, 16, 32 accepted answers |

The runner deliberately limits one command to a small batch. A high tier should
be the result of several observed runs, not one unreviewed burst.

## What not to scale

Do not automate stars, followers, sponsors, reactions, or activity in unrelated
repositories. Those signals represent other people's choices and should remain
organic.
